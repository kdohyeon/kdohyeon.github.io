MSA API 마이그레이션 — 영향도 분석부터 S3 점진적 교체까지

2021 쿠팡

Situation

MSA 환경에서 외부 서비스가 API를 v1에서 v2로 버전업하면서 내부 서비스를 교체해야 했습니다. 단순히 호출 주소를 바꾸는 일이 아니었습니다.

  • 광범위한 의존: API v1의 DTO가 내부 서비스 전체에 걸쳐 사용되고 있어, 교체하려면 대규모 코드 수정이 필요했습니다.
  • 높은 장애 임팩트: 내부 서비스 대부분의 기능이 API v1에 의존하고 있어 장애 발생 시 영향 범위가 매우 컸습니다.
  • 복잡한 DTO 구조: API v1 DTO는 100개가 넘는 필드를 가지고 있었으며, v1과 v2의 필드가 1:1로 매핑되지 않아 데이터 정합성 검증이 필수였습니다.

Behavior

내부 전용 DTO 분리

API v1 DTO를 내부 서비스 전체에서 직접 사용하는 구조를 개선하기 위해, 내부 서비스 전용 DTO를 별도로 만들었습니다. 이렇게 하면 향후 v3, v4로 버전업될 때 외부 DTO → 내부 DTO 변환 레이어만 수정하면 되어 유지보수 비용을 크게 줄일 수 있습니다.

데이터 정합성 모니터링

교체 전, v1과 v2의 응답 데이터를 실시간으로 비교하는 코드를 작성하고 불일치 발생 시 슬랙으로 알럿을 전송했습니다. v2를 제공하는 팀과 미팅을 통해 데이터 차이를 정리하고 일치시키는 과정을 진행했습니다.

public boolean isDifferent(ApiV1Dto v1Dto, ApiV2Dto v2Dto) {
    if (v1Dto.getA() != v2Dto.getA()) {
        return true;
    }
    if (v1Dto.getB() != v2Dto.getB()) {
        return true;
    }
    ...
    return false;
}

S3 Feature Flag로 점진적 트래픽 전환 (1% → 100%)

한 번에 전환하면 위험하기 때문에, S3에 퍼센티지 값을 저장하고 코드에서 동적으로 읽어 v1/v2 호출 비율을 재배포 없이 조절하는 방식을 구현했습니다.

public void abTest() {
    var random = (int)(Math.random() * 100);
    var n = getNfromS3();
    if (random < n) {
        // call API v2
    } else {
        // call API v1
    }
}

Impact

  • 마이그레이션 완료: API v1 → v2로 내부 서비스 전체 교체 성공.
  • 장애 경험과 회복: List 데이터의 값 비교를 누락한 정합성 로직 결함으로 장애가 발생했으나, 롤백·데이터 보정·로직 전면 수정 후 재교체를 완료했습니다.
  • 향후 마이그레이션 비용 절감: 내부 전용 DTO 도입으로 v3/v4 업그레이드 시 수정 범위를 변환 레이어로 한정할 수 있는 구조 확보.
  • 교훈: 데이터 정합성 검증 시 List 요소의 개수뿐 아니라 값까지 비교해야 한다는 것을 실전으로 학습.
