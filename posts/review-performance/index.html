리뷰 API 응답속도 200ms → 30ms, 5단계 개선기

2022 트렌비

프로젝트 개요

개발 기간
—
팀 구성
—
담당 역할
—
Java Spring Boot JPA Redis Pinpoint APM

Situation

트렌비 리뷰 서비스(review-service) 런칭 이후 사용자와 트래픽이 증가하면서 두 가지 시스템 알럿이 지속적으로 발생했습니다.

  • DB 커넥션 풀 고갈: 대량 트래픽 구간에서 "Could not open JPA EntityManager" 에러 반복 발생
  • API 타임아웃: 외부 서비스 의존으로 인한 연쇄 지연

p90 API 응답 속도는 약 200ms ~ 250ms 수준으로, 사용자가 '즉각적인 응답'으로 느끼는 기준인 100ms 이내를 크게 초과하고 있었습니다.

Behavior

원인을 분석하고 5단계에 걸쳐 순차적으로 개선했습니다.

1. n번 API 호출 → Bulk API 1번 호출

주문 서비스(order-service)를 루프로 n번 호출하던 로직을 Bulk 형태의 API로 교체했습니다. 이를 통해 p90 응답속도를 200ms → 100ms 수준으로 단축했습니다.

2. 외부 서비스 의존 제거

주문 서비스에서 가져오던 '구매 옵션' 데이터를 리뷰 저장 시 자체적으로 함께 저장하도록 변경하여 외부 의존을 완전히 끊었습니다. 응답속도가 100ms → 30~40ms로 급감했습니다.

3. 트랜잭션 분리로 DB 커넥션 풀 고갈 해소

@Transactional 메소드 내부에서 외부 API 호출 후 DB I/O를 수행하는 구조가 커넥션 점유를 장기화시키고 있었습니다. 외부 I/O와 DB I/O 구간을 분리하고, DB 로직에만 TransactionTemplate을 명시적으로 적용해 커넥션 풀 고갈을 해결했습니다.

4. Redis 캐시 레이어 도입

상품 시스템의 외부 API가 간헐적으로 타임아웃을 일으켜 장애가 전파됐습니다. 리뷰 도메인이 필요로 하는 상품 정보는 변동성이 낮다는 점을 파악하고 Redis 캐시 레이어를 앞단에 적용했습니다. 스파이크성 에러가 눈에 띄게 줄었습니다.

5. 인덱스 누락 발견 및 추가 (Pinpoint APM 활용)

APM 도구 Pinpoint로 특정 조회 API가 약 600ms를 소요하는 것을 발견했습니다. 중복 리뷰 방지용 단순 조회 쿼리(SELECT * FROM review WHERE order_item_id = ?)에서 order_item_id 인덱스가 누락되어 풀 스캔이 발생하고 있었습니다. 인덱스 추가로 단기간에 큰 폭의 성능 향상을 달성했습니다.

Impact

  • p90 응답속도: 200~250ms → 30~40ms (약 80% 단축)
  • 시스템 알럿 해소: DB 커넥션 풀 고갈 및 API 타임아웃 알럿 더 이상 발생하지 않음
  • 5단계 개선을 통해 단일 수정이 아닌 구조적 원인을 제거하여 재발 방지 효과 달성
