리뷰 서비스 안정화 및 p90 API 응답 속도 개선

2022 트렌비

들어가며

트렌비 백엔드 개발자로서 트렌비 리뷰 서비스 (review-service) 의 성능을 개선하게 된 이야기를 해보고자 합니다.

문제 상황

리뷰 서비스 런칭 후 사용자와 트래픽이 증가함에 따라 아래 두 가지 시스템 알럿이 지속적으로 발생했습니다.

  • DB 커넥션 풀 고갈 이슈
  • API 타임아웃 이슈

또한, 리뷰 서비스의 p90 API 응답 속도가 약 200ms ~ 250ms 수준이었으며 트래픽 증가 시 더 느려지기도 했습니다.

일반적으로 사용자가 '즉각적인 응답'으로 느끼는 한계 시간인 100ms 이내의 응답을 제공하지 못하고 있었습니다.

개선 목표

  • API 응답 속도를 100ms 이하로 단축
  • 리뷰 서비스에서 발생하는 시스템 알럿 해결

개선 이야기. 5가지 과정

1. n번의 API 호출을 1번으로 줄일 수 있을까?

기존 로직은 n개의 주문 ID에 대한 구매 옵션을 조회하기 위해 주문 서비스(order-service)를 루프를 돌며 n번 호출하고 있었습니다.
이를 파라미터로 List를 받아 한 번의 API 호출로 모두 가져오는 Bulk 형태의 API로 변경함으로써 p90 응답 속도를 200ms 수준에서 약 100ms 수준으로 개선했습니다.

2. 굳이 외부 서비스를 호출해야 할까?

주문 서비스에서 가져오는 '구매 옵션' 데이터는 생각해보면 리뷰를 작성할 때 리뷰 도메인 내부에도 함께 저장될 수 있는 데이터였습니다.
리뷰 저장 시 옵션 데이터도 자체적으로 저장 및 관리하도록 변경하여, 외부 서비스 의존성을 끊어내고 API 속도를 100ms 수준에서 30~40ms까지 획기적으로 개선했습니다.

3. "Could not open JPA EntityManager" 에러 해결하기 (트랜잭션 분리)

대량 트래픽이 몰릴 때 DB 커넥션 풀 고갈이 발생했습니다. 원인을 분석해보니 @Transactional이 선언된 메소드 내부에서 지연이 발생할 수 있는 외부 API 호출을 먼저 하고, 이후에 내부 DB 접근을 하고 있었습니다.
메소드 진입 순간부터 커넥션을 점유한 채 외부 응답 대기가 발생하면서 커넥션 풀 고갈이 일어났던 것입니다.
이를 해결하기 위해 외부 I/O와 DB I/O 트랜잭션을 분리하고, DB 로직 구간에만 TransactionTemplate을 명시적으로 적용함으로써 커넥션 풀 고갈 문제를 해결하고 시스템을 안정화했습니다.

4. 성능이 안 좋은 외부 API에 의존하고 있으면 어떻게 해야 할까? (캐시 적용)

리뷰 서비스는 중고 상품 데이터를 조회하기 위해 지속적으로 상품 시스템을 호출했는데, 이 외부 API가 타겟에서 종종 타임아웃 문제를 일으켰습니다.
본 도메인에 필요한 외부의 상품 정보가 주로 변동성이 적은 데이터임을 간파하고 Redis 캐시 레이어를 앞단에 적용하여 매번 호출하지 않고 캐시를 참조하게 만들었습니다. 이를 통해 외부 의존으로 인한 장애 전파 스파이크성 에러를 확연히 통제했습니다.

5. 핀포인트를 활용한 병목 지점 개선 (인덱스 누락)

APM 도구인 핀포인트(Pinpoint)를 통해 특정 조회 API가 예상보다 긴 시간(약 600ms) 소요되는 것을 발견했습니다.
해석된 쿼리는 중복 리뷰 방지를 위한 단순 조회성 쿼리(SELECT * FROM review WHERE order_item_id = ?)였는데, order_item_id에 인덱스가 누락되어 DB 풀 스캔을 돌고 있었습니다.
해당 테이블 컬럼에 인덱스를 추가함으로써 단기간에 큰 폭의 성능 최적화를 적용했습니다.

정리하며

최종적으로 API 응답 속도를 200~250ms에서 약 30~40ms 수준으로 개선했으며 (약 80% 이상의 통신 시간 단축), 간헐적으로 발생하던 무수히 많은 시스템 알럿들도 더 이상 발생하지 않게 되었습니다.
