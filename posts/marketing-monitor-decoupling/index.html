Situation

기존 마케팅 발송 모니터링 시스템은 전사 분석용 데이터 플랫폼인 Hadoop(Hue)에 직접 의존하는 구조였습니다. 이로 인해 다음과 같은 운영상 리스크가 반복됐습니다.

  • SLA 불일치: 분석 전용인 Hadoop은 고가용성을 보장하지 않아, 주말 장애 발생 시 데이터팀 대응이 월요일에 이루어짐 → 주말 발송 현황 파악 불가.
  • 성능 저하: 대규모 분석 쿼리(Heavy Query)가 수행될 때마다 어드민 조회 응답 속도가 급격히 떨어져 운영팀 불편 가중.
  • 구조적 강결합: 데이터 소스의 불안정이 곧 마케팅 운영 중단으로 이어지는 단일 장애점(SPOF) 구조.

Behavior

분석(OLAP)과 서비스(OLTP)의 목적을 분리하고, 각 데이터 성격에 맞는 기술 스택으로 재구성하는 전략을 주도했습니다.

  • 이해관계자 조율: DBA, 데이터팀, 개발팀, 현업 사용자를 아우르는 다자간 협의를 주도하여 "Hadoop 적재는 유지하되, 서비스용 데이터는 별도 관리한다"는 합의 도출.
  • 데이터 소스 분리 (Hadoop → MySQL):
    • 어드민용 실시간 조회 데이터를 MySQL로 이관하여 인덱싱 최적화 및 안정적인 응답속도 확보.
    • 전체 데이터 분석은 기존 Hadoop을 유지하도록 이중 적재 파이프라인 설계.
  • Data Retention Policy 수립: 대용량 로그 처리를 위해 최근 5일치 상세 데이터만 MySQL에 보관하고 이전 데이터는 자동 삭제하도록 DBA 협의 하에 정책화.
  • Batch Statistics 설계: 과거 통계 데이터는 새벽 배치를 통해 요약 테이블로 가공하여 조회 효율 극대화.

Impact

  • 응답속도 80% 개선: 어드민 조회 응답속도 0.5s~1s → 0.1s~0.2s로 단축.
  • 모니터링 가용성 100% 확보: Hadoop 장애 여부와 무관하게 마케팅 운영팀이 24/7 실시간 발송 현황 확인 가능.
  • 장애 전파 리스크 제거: 시스템 간 강결합 해소로 긴급 장애 대응 피로도 대폭 감소.
